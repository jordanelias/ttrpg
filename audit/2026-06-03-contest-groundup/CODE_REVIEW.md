# Ground-Up Contest — Code Architecture Review (granular)

`[SELF-AUTHORED — bias risk]` My own committed code, read line by line and attacked. Every finding cites the actual line/behavior. Verdict first, worst first.

## Verdict
The architecture is right in **intent** (engine / primitives / resolver / modes / policy separation; decoupled policy) but the **execution leaks** in ways that matter. Two are functional, not stylistic: a stated core primitive does nothing, and the foundation has a circular dependency papered over by a lazy import. The engine module is the one clean, well-bounded piece — fixes need not touch it.

## P1 — a core primitive is inert
- **Appeals do nothing.** `Move` carries `appeal` (resolver.py:19), policies set it (`honest`→LOGOS, `overreacher`→PATHOS, policy.py:29/50), but `_process` and `_reception` (resolver.py:45–69, 39–43) **never read `mv.appeal`**. The three modes of proof — a primary primitive (the `Appeal` class, "the per-move axis") — have zero mechanical effect. The class + the policy plumbing imply it works; it doesn't. Either appeals drive resolution (ethos→standing, pathos→audience, logos→merits, as the framework says) or the field and the class are decoration.
- **Self-gating is mostly unrealized.** `SelfGating.hard_device_licensed` is consulted on exactly one branch — the `'hard'` move (resolver.py:60). The corpus triage (opponent × audience → the licit *tactic set*) is collapsed to a single boolean for a single move-kind. The primitive's surface is far narrower than its docstring (primitives.py:47–49) claims.

## P2 — foundation and resolution-loop defects
- **Circular dependency, hidden by a lazy import.** `policy.py:7` imports `Move` from `resolver`; `resolver.py:72` imports `ContestView` from `policy` — but inside `run()`, lazily, to dodge the cycle. The two layers are mutually dependent. `Move` and `ContestView` are the *contract* between them and belong in their own module (e.g. `contract.py`) that both import. As written, the layering claim is muddled at the base.
- **`_opp_standing`: a hidden side-channel with a bug-masking default.** `run()` injects `c._opp_standing` onto the contestant (resolver.py:79); `_process` reads it via `getattr(c, "_opp_standing", 5.0)` (resolver.py:61). It is not in `Contestant.__init__`, so any other caller of `_process` silently gets opponent-standing **5.0** — a magic default that hides errors. The opponent (or its standing) should be an explicit `_process` parameter; the signature (resolver.py:45) is missing it, which is *why* it reaches for a stowaway attribute.
- **`_process` is a god-function with inconsistent state flow.** Lines 45–69 do eight things — committed-ground init, affordability/yield, reserve spend, support, shift-legality + contradiction, barred check, evasion, reception + merits + standing. One piece of state (`live_ground`) is threaded **functionally** (returned); the other four (`state`, `reserve`, `standing`, `merits`) are **mutated in place**. That mixed style is exactly where the turn-order bug lived (merits mutated and checked mid-loop). Move-kind dispatch and the legality checks and the scoring effects should be separated.
- **Fault state has no single definition, and precedence is written twice.** The schema `{evasion, yields, contradicted, barred}` is declared implicitly in three places — `Contestant.state` (resolver.py:28), `DefeatConditions.check` (primitives.py:82–88), `_why` (resolver.py:95–99). Worse, `check` returns the losing side but **discards which fault fired**, so `_why` re-scans the same dict and **re-implements the same precedence** (evasion/yields thresholds appear in both). `check` should return `(side, reason)` once; the state should be one owned dataclass.
- **Stringly-typed side identity, everywhere.** `"a"`/`"b"` are magic strings threaded through `Merits` (the `setattr(self, side, …)` / `getattr` in primitives.py:68 — a typo silently creates a new attribute instead of erroring), the `state` dict keys, the `run()` loop tuples, and `_why`. No single definition of "a side."

## P3 — placement, consistency, validation, tests
- **Mode differentiation is cosmetic.** `FLAVORS` (modes.py:16–19) separates deliberative from forensic only by `start_ground`; `base_ob` is `2.0` in both, so it is redundant in the dict. The corpus's distinct *win conditions* (decide future action vs. verdict-on-past-act with a proof-hierarchy) are **not in the code** — same engine, different opening ground.
- **Stateless class instantiated per roll; static/instance split is inconsistent.** `_reception` does `Leverage().net(...)` every call (resolver.py:40) though `Leverage` holds no instance state. Its stateless siblings (`Stasis`, `SelfGating`, `DefeatConditions`) use `@staticmethod`; `Leverage` uses an instance method. Pick one.
- **The pool model is an orphan.** `pool_of` / `POOL_BASE` (resolver.py:12–14) is a core mechanical `[SEED]` living as a loose function in the resolver while every other quantity is a primitive class; `_reception` even calls `pool_of(c.faculty)` twice (lines 41–42).
- **`Config` conflates contest shape with the adjudicator.** threshold/budget/base_ob (the contest) sit beside `audience_learned`/`audience_hostile` (who judges) — resolver.py:30–37. The adjudicator is its own entity in the framework and won't fold into `Config` cleanly when ceremonial/dyadic arrive (ceremonial *is* the audience; dyadic has none).
- **No boundary validation.** `Move("garbage")` reaches `Reserve.can` → `self.COST[kind]` **KeyError** (primitives.py:41), a deep crash instead of a clear rejection; `Stasis.LADDER.index` (primitives.py:18) raises `ValueError` on an unknown ground; `reserve_frac` divides by `reserve.max` (resolver.py:83) with no zero-guard. Nothing validates `Move.kind` / grounds at the boundary.
- **Dead imports.** `primitives.py:6` imports `dataclass, field` and uses **neither** (its classes are plain). `resolver.py:8` imports `field` and never uses it.
- **Duplicated default.** `ContestedMode.play` defaults `standing_a=5.0` (modes.py:24) while `Contestant` uses `Standing.START` (=5.0); two sources for one value — they drift if `START` changes.
- **The orchestration is only integration-tested.** Unit tests hit the primitives directly and the modes end-to-end, but `_process`/`run` — the most defect-prone code, where the turn-order bug lived — have **no unit test for their invariants** (e.g. symmetry). That bug was caught by a stress test precisely because no unit test asserted fairness.

## What this implies for the pending fix
The resolver fix should not be a patch on top of this. Folding the turn-order fix (boundary resolution) in is the moment to: extract a `contract.py` (Move + ContestView + a `Side` and a `FaultState` dataclass), make `_process` take `opp` explicitly and split it into dispatch + scoring, give `check` a `(side, reason)` return (delete `_why`), make appeals actually drive resolution (or remove the field), move the pool model into a primitive, and add unit tests for the orchestration invariants. None of this touches `engine.py`.

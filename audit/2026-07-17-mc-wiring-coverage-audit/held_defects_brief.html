D3 Parliamentary Mandate restoration code verifiedSC lane
Executive summary

When a parliamentary vote hits Total Victory, the losing side's dominant faction takes Mandate −1 — the code calls it a one-season penalty, but the restoration was never written, so it was permanent. Total Victory fires ~7×/campaign, so factions quietly bled Mandate for the whole game. Wiring the promised restoration is correct and verified, but handing that Mandate back flips ~72% of campaign winners and breaks the seeded golden — so it's a balance change, not a routine fix.

The bug

In parliamentary_vote.py §10, when the persuasion track reaches Total Victory (≥ 9) or Total Defeat (≤ 1), the losing coalition's dominant faction (highest Mandate/L) takes adjust("L", −1). A note beside it reads: "[one-season penalty; temporary-modifier restoration deferred to season_manager]". But season_manager never implemented any restoration. The −1 is therefore permanent — an invisible, unintended Mandate drain.

Total Victory→loser Mandate −1→(promised restore never runs)→permanent

The fix (implemented + verified)

  • At the penalty site, record its exact reversal on the world: {faction, +delta, applied_season}.
  • season_manager.advance_season restores it exactly one season later, reversing the granular delta (so it preserves any other Mandate change since — it's a temporary modifier, not a value snapshot).
  • Verified over a full campaign: 7 penalties applied → 7 restored, one-season lifetime, nothing lingering.
  • Scoped strictly to this resolver's penalty — it does not touch the separate faction_layer §5.4 Censure penalty, whose compounding with this one is the still-open ED-SC-0015 stacking question.

Impact & why it's held

Over a deterministic 40-seed A/B, restoring the Mandate shifts wins to the Church (+4) — it was most often on the losing side of these votes, so it bled the most — away from Crown (−2) and Hafenmark (−2). Varfell's total is flat, yet individual Varfell seeds still flip: 29 of 40 campaigns (72%) change winner. That breaks 7 parliamentary_bridge golden tests (they pin the win-share). Correct behaviour, real balance shift — needs a golden re-baseline + your sign-off.

D6 Populate the NPC / knot / settlement registries not builtWR / IN lane
Executive summary

Three generators — generate_npc, form_knot, register_settlement — have zero callers, so world.npcs / knots / settlements are always empty and every consumer that reads them silently no-ops every season. The registries and their logic are real and tested; they just never receive inputs. Populating them is the fix — but it needs a spawn policy, and because generation draws from the shared RNG stream, it perturbs strategic outcomes, so it lands with the balance work, not the bounded set.

The bug — "wired but vacuous"

create_world never calls the generators, so e.g. simulate_npc_actions (which drifts NPC stances toward each other on a volatility check) iterates an empty dict and returns nothing, every season. Same shape for world.knots and world.settlements.

generate_npc (0 callers)→world.npcs = {}→simulate_npc_actions no-op

Three heterogeneous parts

PartWhat populating needsDesign ambiguity
SettlementsFeed register_settlement from the canonical ~35-settlement geography roster at world creation.Low — the roster is canonical data.
NPCsgenerate_npc builds one via two-tier ecology weighting; spawning needs a count and cadence (per territory? at creation vs per season?).Medium — no single canonical count.
Knotsform_knot(a,b) binds two NPCs — but when a knot forms needs a trigger/threshold, and depends on NPCs existing first.High — least specified.

The catch (why it's balance-affecting)

generate_npc draws from world.rng — the same stream strategic resolution uses. Injecting generation calls shifts every subsequent random draw, changing the strategic trajectory → different winners. (Unlike D2's inert IP add, which never touched the RNG.) To keep it strategy-neutral, NPC generation should use its own RNG; otherwise it re-baselines the golden like D3. I'd bring you a conservative spawn proposal before wiring it.

D1 Turmoil — the write-dead victory clock not builtWR lane
Executive summary

Victory requires "Political Stability ≤ 6," read off the Turmoil clock — but nothing writes Turmoil, so it sits at 0 forever and the stability gate is trivially always true. A peninsula in total chaos still passes the check. The canonical mechanic (peninsular_strain §4) is a real accounting-time computation, so wiring it faithfully makes the gate meaningful — and shifts balance. Plan: do §4.1/§4.2 now (the clock moves, the gate becomes real); flag §4.3's threshold effects as a bigger later layer.

The bug

Turmoil initialises at 0 and is read once by victory.py (PS = clocks['Turmoil']; PS ≤ 6), but no live module writes it — every writer lives in deprecated / old-sim code. So one of the three victory sub-conditions is a permanent no-op.

The canonical mechanic — peninsular_strain_v30 §4 (CANONICAL, ED-743)

§Rule (evaluated at each Accounting)
4.1 advance+1 per controlled territory at Accord ≤ 1 (cap +3/season) · +2 on a faction elimination · +1 per territory revolt. (Held-instability, not battle-occurrence.)
4.2 decay−1 if no instability advanced this season · −1 per active treaty pair (cap −2) · −1 for a public diplomatic resolution.
4.3 thresholds3–4 Tension: all factions Legitimacy −25 · 5–6 Fracture: Accord −1 somewhere · 7–8 Crisis: Accord −1 all non-capitals + Mandate check · 9–10 Collapse: Accord cap 2, MS −1/season.

Plan & wiring subtleties

  • Implement §4.1/§4.2 at the accounting step (alongside CI/MS) — that makes the clock move and the gate real. Flag §4.3 (the downstream Legitimacy/Accord/MS effects) as a separate, larger balance layer.
  • Accord is stored as granular floats (via ACCORD_MAP), so "Accord ≤ 1 / ≥ 2" needs band-mapping, not a raw compare.
  • Treaty-pair decay reads world.treaties — currently empty (that's D6), so that decay source contributes 0 until treaties are populated. Honest partial faithfulness.
  • Canon here is unambiguous (unlike D6/D8), so this is the most "just implement it faithfully" of the balance set — pending your go + a golden re-baseline.
D8 The Character/Actor dataclass & the roster fork needs a rulingPC / IN lane
Executive summary

There is no Character/Actor class anywhere — the attribute→derived-stat foundation every personal-scale roll needs has no runtime home, so resolvers duck-type getattr(actor,'strength',3) or bypass character state. It's the single biggest structural hole (and it blocks every personal-scale Godot port). Worse, two rival attribute rosters coexist, so building the class forces a schema decision first — which is why this one needs your ruling before any code.

The gap

World has Faction, Territory, and registry dicts — but no class for a person. Combat, contest, and fieldwork resolvers fake attributes via getattr defaults or skip character state entirely. Nothing holds a character's nine-or-ten attributes, so no derived stat (Health, Stamina, Composure…) has a real source.

They disagree on the Mental group (Cognition + Recall separate, vs. merged into Acuity) and on Spirit (a standalone 10th, vs. renamed Will inside Mind).

The fork — why it's yours

The 10-roster is the de facto live schema: the one working port and the derived-stat formulas depend on it right now. The 9-roster is the de jure intended consolidation — but unsettled (IN FLUX, [ASSUMPTION]s, unwired aggregates). Building the dataclass on the 9-roster mismatches the live formulas; building on the 10-roster commits to the schema the registry means to retire.

My recommendation

Ratify the descriptor_registry 9-consolidation as the go-forward roster (turn its [ASSUMPTION] merges — Cognition→Acuity, Spirit→Will — into a ruling), and build Character on it with a thin compat layer that maps the live derived-stat inputs (Spirit→Will, Cognition→Acuity) so combat_engine_v1 keeps working unchanged during the transition. That settles the schema once — the registry is meant to be the single source of truth — instead of cementing the roster the corpus is trying to leave. Lower-friction alternative: build on the 10-roster now and migrate later (re-opens the consolidation).

Either way: once you pick a roster, I build the dataclass + point the resolvers at it (retiring the getattr duck-typing). That's the architectural pass — separate from the balance PRs above.

# Batch 9: Mass Battle Engine Terms and the Coherence/Cohesion Near-Collision

**Methodology note (applies to all ten terms below):** The mass-battle corpus is enormous — grepping any of these terms against the full repo returns 150–250+ files (the tool's result cap), the overwhelming majority of which are session-narrative audit folders (`designs/audit/*`), frozen historical sim-run logs (`tests/sim/*.md`, predating the `sim/` package per `sim/README.md`'s own disambiguation), and `archives/`/`deprecated/` material that CLAUDE.md §1 already tells us is not canonical. For each term I read the canonical mechanical corpus in full (the named primary sources, `params/`, `references/*.yaml`+`glossary.md`, `canon/`, the live engine `tests/sim/mass_battle/*.py`, the separate `sim/provincial/*.py` armature, `tests/valoria/*.py`, `CURRENT.md`, `handoffs/HANDOFF_MB.md`) and then opened a representative sample of the audit/archive hits per term to confirm they *cite or discuss* the mechanic rather than independently restate/redefine it. Where that pattern held I say so rather than pasting hundreds of identical citations; where it didn't (a genuine restatement site), I open and quote it. This is a deliberate completeness-vs-tractability trade-off, disclosed per your instructions rather than silently applied.

## 1. Size (mass battle)

**Home definition.** `designs/provincial/mass_battle_v30.md:24` — "**Size** — headcount. Health pool. Casualties reduce it directly." Full mechanical definition at `mass_battle_v30.md:140`: "**Size** — computed from TroopCount: `Size = floor(TroopCount / block_size)`. At Size 0 (TroopCount < block_size): destroyed." Scale table at `mass_battle_v30.md:70–88` (§A.3): `block_size` varies by battle scale — Skirmish=10, Company=100, Battle=500, Campaign=1,000, War=5,000 — and `TroopCount = Size × block_size` is "set at muster." Range: integer, 1–7 nominal stat but *computed*, not freely assigned. Glossary home: `references/glossary.md:123` (Part Five) — "Size — Unit headcount stat (replaces Strength). At 0: unit destroyed. Pool contribution capped at Command. (PP-232)."

**Live-engine home:** `tests/sim/mass_battle/hierarchy/units.py:1486` — `self.size = max(1, total // TROOPS_PER_SIZE)` and `:1541–1545` `recalc_size()`: `effective_size = hp / BLOCK_SIZE`, `size = floor(effective_size)`. `TROOPS_PER_SIZE`/`BLOCK_SIZE` both = 100 in `tests/sim/mass_battle/config.py:19,40`, the latter annotated `# [canonical: designs/provincial/mass_battle_v30.md §A.3 — Company scale]`.

**Exhaustive usage sweep.** `Size` (as the mass-battle stat, filtered from ordinary-English "size") appears canonically/consistently in: `designs/provincial/mass_battle_v30.md` (dozens of times, e.g. lines 24, 32–34, 70–88, 140, 190–220, 410–460, 613–662, 711); `designs/provincial/mass_battle_v30_infill.md`/`_index.md` (co-file pair, mirrors the head); `params/mass_combat.md:66–110` (Definitions/Unit Stats tables); `references/glossary.md:123`; `references/values_master.yaml:1621–1907` (worked examples, "Size 5, Command 5..."); `canon/mechanics_index.yaml` (mass_battle entry); `designs/provincial/military_layer_v30.md:36–103` (BG-layer unit stats — its own "Size" column, target-size table); `designs/provincial/mass_battle_integration_v30.md`; `sim/provincial/massbattle.py:100–107,714,954–1907` (bare-port copy) and `sim/provincial/units.py:322,342` (`hp_max = size_max * BLOCK_SIZE`); `tests/sim/mass_battle/{config,hierarchy/units,orchestration,core/*}.py` (engine, extensively, e.g. `units.py:1461–1494,1541–1550`); `tests/valoria/test_mass_battle_maneuvers.py`, `test_mass_battle_byte_exact.py` (exercise-only, no independent restatement). The large `designs/audit/2026-0*` / `tests/sim/sim_mb_06_v*.py` / `archives/*` hit-set (150+ files) is either superseded engine snapshots (v5 through v25, all predating the current package) or audit prose citing the same formula — spot-checked `designs/audit/2026-06-30-massbattle-bottomup/00_inventory.md`, `designs/audit/2026-07-04-ners-qualitative-audit/01_workings/dossiers/dossier_mass_battle.md`, and three `sim_mb_06_v*.py` snapshots; all cite/derive from the same `floor(TroopCount/block_size)` rule without inventing a different one.

**Divergence check.**
- **Scale-table scope gap (engine implements only one of five design-doc scales):** `mass_battle_v30.md:70–88` offers five selectable `block_size` values (10/100/500/1,000/5,000) keyed to Skirmish/Company/Battle/Campaign/War. The live engine hardcodes a single value: `BLOCK_SIZE = 100` / `TROOPS_PER_SIZE = 100` (`config.py:19,40`), annotated as "Company scale" only. There is no runtime toggle or parameter selecting Skirmish/Battle/Campaign/War block sizes anywhere in `tests/sim/mass_battle/` (confirmed by grep across the whole package — `BLOCK_SIZE`/`TROOPS_PER_SIZE` are referenced only as the fixed constant, never scale-selected). The design doc's five-scale table is therefore **unimplemented outside Company scale** — not a numeric contradiction, but a real implementation-coverage gap between the documented mechanic and what the engine can actually produce.
- **`sim/provincial/massbattle.py` is a frozen, older duplicate.** Its own docstring (`massbattle.py:1–35`) says it is a "**bare port from tests/sim/sim_mb_06_v22.py 2026-05-18**" — i.e., a snapshot of the mass-battle engine as it stood at v22, before the current `tests/sim/mass_battle/` package's ED-899 (Command-only pool), ED-1013 (smooth cohesion pool), ED-1019 (per-subunit rout/Morale/Discipline), and ED-1088/1089/1090 (LC-8 shape retirement, field-movement default, subunit cap 11) all landed. Concretely for Size: `sim/provincial/massbattle.py:100–107` still documents the byte-exact-with-v22 formula (`Pool = min(Size,Command)+Command`) as *the* formula with no ED-899/ED-1013 override, and its geometry constants (`BATTLEFIELD_SIZE=25`, `UNIT_GRID_SIZE=15`, `sim/provincial/massbattle.py:54,56–57,60–61`) predate even the *design doc's* own superseded 41-grid sketch (mass_battle_v30.md §A.3b), let alone the current engine's 50-grid (`config.py:9–13`). This is a **hardcoded duplicate that has already drifted** from current canon on multiple axes (pool formula, battlefield geometry, and the entire per-subunit model, which `sim/provincial/units.py` has no trace of at all).

## 3. Discipline (mass battle)

**Home definition.** `mass_battle_v30.md:169` — "**Discipline** (1–7) — organisational integrity. Starting value = min(general's Command, Military ceiling above)." Degradation rule §A.4:172–188 (deterministic, two-condition trigger). `references/glossary.md:125` — "Discipline — Unit organisational integrity (replaces Cohesion, PP-232). Contribution to Health per Size capped at Command."

**Exhaustive usage sweep.** Extremely widely used — canonical/consistent throughout `mass_battle_v30.md` §A.4–A.14 (dozens of citations, e.g. per-sub-unit model ED-1019/1020/1022/1026/1027 at `:216–226`), `params/mass_combat.md:106–170,335–349,464–470`, `references/glossary.md:125`, `designs/provincial/military_layer_v30.md:36–150` (own BG Discipline degradation-from-Wealth-Zero rule), engine `tests/sim/mass_battle/hierarchy/units.py` (`Subunit.discipline`/`eff_discipline`/`degrade_discipline`/`restore_discipline`, `:280,388–389,456–469`) and `Unit.discipline_penalty` (`:1551–1555`), `core/exchange.py:55–68` (per-subunit pool contribution). `sim/provincial/massbattle.py:183–190,277,1329–1334` (bare-port duplicate).

**Divergence check — this is the richest finding in the batch.**

**(a) The pre-PP-232 "Cohesion" name is still live as a stale section heading in a canonical params file.** `params/mass_combat.md:3` documents the rename banner: "PP-232: Unit stats renamed (Strength→Size, Combat Power→Power, **Cohesion→Discipline**, Coherence Rating/Command Rating→Command)." Yet two section *headings* in the same file were never swept:
- `params/mass_combat.md:335` — `### PP-222: Offensive Lock Blocks Cohesion Degradation [PROVISIONAL]` — body text (`:336`) correctly says "blocks Discipline degradation checks," but the heading still says "Cohesion Degradation."
- `params/mass_combat.md:347` — `### PP-231: Cohesion Degradation = Asymmetry Mechanic [PROVISIONAL]` — body text (`:348`) correctly says "Discipline degradation is an asymmetry mechanic," heading again says "Cohesion Degradation."

Both are residual, unswept post-rename artifacts: the file's own top banner documents the Cohesion→Discipline rename, but two of its own subsection titles never got the memo.

**(b) `mass_battle_v30.md` itself uses "Discipline" for two different mechanics in its own two Parts.** Part A (TTRPG, §A.4) "Discipline" = the per-unit 1–7 organisational-integrity stat described above. Part B (Board Game, §B.3) uses "Discipline" for something else entirely: `mass_battle_v30.md:770` — *"Attacker Discipline −15 (renamed from Cohesion per PP-232; **Discipline is the derived stat = Stability × 10 per derived_stats_v30 §8**). ED-705."* This is a **faction-level derived value** (range 10–70, `= Stability × 10`, confirmed at `designs/scene/derived_stats_v30.md:301`: `| Stability | Discipline | Stability × 10, starting = stat × 10 | Internal faction unity |`), not the mass-battle per-unit organisational stat at all. So within one canonical head, "Discipline" denotes: (i) a per-unit 1–7 combat stat (Part A) and (ii) a faction-wide 10–70 derived economic/political value (Part B) — two genuinely distinct mechanics sharing one name, in the *same document*.

**(c) This exact collision was flagged by a prior audit and remains unresolved in current canon.** `designs/audit/mass_battle_interdependency_2026-04-29.md:55–61` (finding [INTER-12b]) says verbatim:
> `"Discipline" name collision: unit stat vs faction derived value` — `derived_stats §10.2 Army Morale: "Discipline modifier: +1 if faction Discipline ≥ 75% of max."` `"Faction Discipline" = Stability × 10 (range 10–70, derived from §8).` `mass_battle: "Discipline" = per-unit organisational stat (1–7, tracked on unit tokens).` `Army Morale formula uses "Discipline" to mean two different things in adjacent bullet points.` … `Recommend: rename faction-level derived value to "Cohesion" or "Unit Readiness" to avoid collision.`

I re-read the *current* `derived_stats_v30.md:405` (quoted in full under Army Morale below) and the collision is **still present verbatim** today — the "Discipline modifier" bullet in the Army Morale composite still reads faction-Discipline (10–70), sitting next to "average unit Morale" which reads per-unit Morale (1–7). The audit's own suggested fix — renaming the faction value to "Cohesion" — was never adopted, and would in any case have collided with the *other* live senses of "Cohesion"/"cohesion" documented in §7 below.

## 4. Morale (per-subunit)

**Home definition.** `mass_battle_v30.md:190–229` — "**Morale** (1–7) — rout threshold. Starting = general's Command + unit quality modifier (cap 7)." Degradation triggers, −3/Cascade-Phase cap, encirclement exception (PP-683/MB-08), floor=1 while general present, rout at 0. **Per-sub-unit model** (ED-1019, 2026-06-17, `:216–226`): each sub-unit tracks its own Morale, routs independently; **continuous, not rounded** (ED-1024, `:222`): "Live Morale is never rounded to an integer." Unit-level rout is *derived*: "a unit routs when its troop-weighted **aggregate** Morale reaches 0, OR all its sub-units have routed, OR its general is gone" (`:218`).

**Exhaustive usage sweep.** Extensive and consistent through `mass_battle_v30.md` §A.4/§A.12/§A.13/Part C/Part D (e.g. `:190–229,591–608,867–869,921–923`), `params/mass_combat.md:112,157–171`, `references/glossary.md` (Morale is *not* a Part Five row — glossary's mass-combat table omits Morale entirely, see divergence below), engine: `tests/sim/mass_battle/hierarchy/units.py` — `Subunit.morale/morale_start/eff_morale/eff_morale_start/erode_morale/pull_morale` (`:281–282,391–397,431–455`), `Unit.agg_morale/derive_rout/cascade_morale_hit` (`:1514,1518–1539`); `config.py:83–93` (`MORALE_FIX`, `MORALE_EROSION_DAMP=0.7`, `MORALE_SIBLING_PULL=0.15` — the ED-MB-0002 DG-4 sibling-pull ruling, `handoffs/HANDOFF_MB.md`); `core/state.py` (morale_check_phase). `sim/provincial/massbattle.py:147,259–264,1479–1541,1663–1702` (bare-port, unit-level-only Morale — no per-sub-unit tracking at all, since it predates ED-1019 by roughly a month).

**Divergence check.**
- **Glossary omission.** `references/glossary.md` Part Five (Combat Terms, `:118–128`) lists Combat Endurance, Command, DR, Size, Power, Discipline, Health per Size, Total Health — **but not Morale**, despite Morale being one of the four core per-unit mass-battle stats named in `mass_battle_v30.md:138` ("UNIT STAT BLOCK (all 1–7)": Size, Power, Speed, Discipline, Morale). This is a real coverage gap in the supposedly-authoritative "canonical reference for all term expansions project-wide" (glossary.md:4) — Morale simply isn't there to diverge *from*, which I flag as its own finding (a term with a real mechanic and no glossary entry) rather than a value contradiction.
- **`sim/provincial/massbattle.py` has no per-sub-unit Morale at all.** Its `Unit`/`Subunit` dataclasses (`sim/provincial/units.py`) carry only unit-level `morale`/`morale_max`-style fields (no `routed`/`broken`/`eff_morale`/`pull_morale` per subunit) — it is frozen at the pre-ED-1019 model. Anyone reading `sim/provincial/massbattle.py`'s Morale-cascade code (`:1479–1541,1663–1702`, its own docstring citing "§A.12 — Morale Cascade") would get the *unit-level-only* mechanic mass_battle_v30.md explicitly says is now superseded by the per-sub-unit layer ("the per-sub-unit model is the granular layer beneath them, not a revision of them," `mass_battle_v30.md:218`) — i.e. this file is not merely duplicate, it silently represents an **earlier design generation** as if current.
- **Continuous vs discrete presentation.** ED-1024 (`mass_battle_v30.md:222`) is explicit that live Morale is a float, never rounded — but the unit stat block header (`:138`) still frames Morale as "(all 1–7)" and the BG unit table (`§B.2:699–709`) prints integer Morale values per unit type. This is not a contradiction (the design doc itself reconciles it: integers are the *starting* values, continuous in play) but is worth flagging as the kind of "same section, two granularities" texture a reader could mis-cite if they only see the header table.

## 5. Army Morale

## 9. Cell troop density

**Home definition — split between a qualitative prose description and unexplained engine constants.** `mass_battle_v30.md:121` (§A.3b): "**Cell troop counts.** Each cell within a formation holds a player-assigned troop count with min/max bounds. **Exhaustive usage sweep.** `CELL_FLOOR`/`CELL_CAP`/`SUBUNIT_ROUT_FLOOR`/`MAX_TROOPS_PER_UNIT` occur only in: `tests/sim/mass_battle/config.py:20–28` (definition), `geometry.py:59,89–91` (`footprint_for` consumption), `hierarchy/units.py` (indirectly, via `troop_count`/`cell_troops`), `core/attrition.py`/`core/contact.py` (density-scaled Lanchester terms, `LANCHESTER_DENSITY_REF=100` at `config.py:145`), `percell.py:21` (`density / BLOCK_SIZE`), `PC_VOLLEY_DENSITY_*` constants (`config.py:56–59`, target-formation-density scaling for Volley), and the narrative-only citation trail: `tests/coverage_matrix_archive.md:230–231,272,276,289,294,412,606` and `tests/coverage_matrix.md:249,251,353` (dev-log entries documenting when/why these constants were introduced or changed). The constant remains defined and exported (`config.py:7`'s `__all__` list includes it) with a comment describing a behavior it does not perform — a stale, misleading, but still-live dead stub.

## 10. Subunit facing/shape

**Home definition (shape).** `mass_battle_v30.md:336` (ED-909, "Jordan-adjudicated 2026-06-09, LC-8 executed 2026-07-02"): *"The geometric formation set is ratified as — **Subunit shapes:** Line, Arrowhead/Wedge, GappedLine, Column; **Unit-level manoeuvres:** Envelopment (Cannae), Refused Flank (Leuctra/Leuthen). **Home definition (facing).** No single canonical prose home — `mass_battle_v30.md:123` (§A.3b, "Sightline," 2026-05-15) is the *only* design-doc statement about facing at all: "every cell has a 135° forward sightline arc from its facing direction, with maximum perception distance of 15 cells... a Line cell facing forward does not turn to face Horseshoe wings that have wrapped behind it because those wings are in the cell's blind spot." **Divergence check.**
- **Two coexisting, numerically different facing/arc models, only one of which is in the ratified design-doc head.** The design doc's own **Sightline** system (`mass_battle_v30.md:123,126–128`) specifies a 135° total forward arc (`SIGHTLINE_ARC_HALF=67.5°`) governing *perception* (can a cell see an enemy at all). The engine's separate **octagon/ANGLE_DEF_MOD** system (`config.py:70–75,127–128`) specifies a 210° total visible arc (`FOV_HALF_DEG=105°`, i.e. `180° − REAR_BLIND_DEG/2 = 180 − 75`) governing a *different* thing — the defensive dice-penalty for being struck from an unfaced angle (GREEN/YELLOW/RED). These are not logically contradictory (one gates perception, the other gates combat-angle defense) but they use the *same vocabulary* ("facing," "blind arc," "FOV") with **different numbers for what sounds like the same physical cone**, and only the first (Sightline) has a home in the ratified `mass_battle_v30.md` head — the octagon/ANGLE_DEF_MOD system's only prose source is `designs/audit/2026-05-31-percell-combat/movement_model_design.md`, a working audit document under `designs/audit/`, never promoted into `mass_battle_v30.md` itself. A reader relying solely on the canonical head would not learn the octagon combat-facing system exists at all, nor that its arc width differs from the Sightline arc it superficially resembles.
- **The design doc self-flags an unreconciled vocabulary split between two "formation" concepts, which I confirm is still open.** `mass_battle_v30.md:336` (the same ED-909 note that defines the shape roster) states: *"The mapping between this A.6 **dice-modifier** formation set (Shield Wall, Wedge, Skirmish, Column, Feigned Retreat, Reserve) and that **geometric** set is an orthogonal-axes question **PARKED as an ED-909 follow-up**."* I confirm this is still unresolved as of the current text — §A.6 (`:326–334`) lists Line/Shield Wall/Wedge/Skirmish/Column/Feigned Retreat/Reserve as dice-modifier "Formation Types," a vocabulary that only partially overlaps the geometric Subunit-shape roster (Line, Arrowhead, GappedLine, Column) — "Wedge" in §A.6 is explicitly equated to "Arrowhead" by the ED-909 note's own phrasing ("Arrowhead/Wedge"), but "Shield Wall," "Skirmish," and "Feigned Retreat" (§A.6, dice-modifier axis) have no counterpart at all in the four-shape geometric roster, and "GappedLine" (geometric) has no counterpart in §A.6. This is a real, currently-open terminology gap, but I flag it as **already self-disclosed** by the design doc rather than a newly-discovered contradiction — the doc itself says the mapping is parked, not settled.
- **`sim/provincial/units.py` still carries the retired Horseshoe/RefusedFlank shapes.** `sim/provincial/units.py:205–230` (its `shape_type()`/geometry logic) explicitly handles `self.shape == "Horseshoe"` and `self.shape == "RefusedFlank"` as live Subunit-level shapes (`:213,224`) — the exact two shapes ED-909/LC-8 retired from the Subunit level on 2026-07-02 in the current engine. This frozen bare-port therefore doesn't just lag on formula values (as with Size/Command/Discipline above) but actively represents **a shape taxonomy the current engine no longer recognizes as valid** at the Subunit level.

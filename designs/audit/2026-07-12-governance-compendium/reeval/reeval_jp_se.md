### SE-JP1 — Goningumi collective-liability cells

- **Original mechanic:** New Keep Order verb ("Bind the Cells") binding a settlement's Local Actors into fixed five-household collective-liability cells — cheap, no PS cost, auto-detection of infractions — with a compounding "Cell Revolt" failure mode if squeezed too hard. Landed as `governance_play_redesign_v1.md` §1.3b (ED-SE-0020), plus a §1.6 Ledger tag variant ("Collective Liability") and a §2.3 Crisis-card trigger ("Cell Revolt").
- **Prior NERS verdict:** **MERGE (confirmed).** Two independently sufficient grounds: (1) fatal substrate incoherence — the mechanic requires partitioning Local Actors into groups of five, but `settlement_layer_v30.md` §4.5 states each settlement generates only **1–2** Local Actors (0 for Outposts); it cannot execute once, on any settlement, as written. (2) Conditional Omega-d dominance — a player self-selects Bind-the-Cells in already-calm settlements, harvesting the benefit while the compounding-risk clause rarely fires.
- **Re-eval vs built work:** **RECONCILE** — named collision between two *different* built/queued representations of "Local Actor," neither of which is the design-prose §4.5 1–2-cap the audit's fatal-flaw finding relied on. `sim/territory/registry.py`'s actual `npc_ids` field is an **unbounded list — no actor cap at all**, and `goldenfurt_slice/` (the one fully-worked Town) runs **9 named actors** for a single settlement, already several times past the §4.5 prose cap the MERGE cited as fatal. Meanwhile `settlement_mgmt_stress_01` represents Local Actors as **count-per-type, not named individuals** — a third, incompatible shape. The audit's "non-executable, category error" finding is true of the §4.5 prose doc but **not demonstrated true of either built representation**; against `npc_ids` (unbounded) or Goldenfurt's 9-actor precedent, a five-household partition is straightforwardly executable. Separately, the audit's Omega-d concern (self-selected low-risk activation, rare compounding payoff) has a directly on-point built precedent: `goldenfurt_slice`'s G606 finding already solved an isomorphic compounding-punishment spiral (the recall death-spiral) via a Submit-to-audit escape valve plus a hard **suspicion cap of +1/season**. Cell Revolt's escalation clause was never checked against that pattern.
- **Changes verdict? Yes.** The MERGE's stated *reason* (category error against a 1–2 actor cap) does not survive contact with either built substrate; the verdict should not remain a flat MERGE on that rationale. It should not simply flip to KEEP either, because the built substrate itself is now three-way inconsistent (unbounded list / 9-actor worked case / count-per-type stress harness) and SE-JP1 was never authored against any of them.
- **Consolidated recommendation:** **reconcile-first.** Before re-promoting §1.3b: (1) declare which Local-Actor representation SE-JP1 targets — `npc_ids` (unbounded, supports literal 5-cell partition), the Goldenfurt named-actor precedent, or the stress-harness count-per-type abstraction (which would require re-deriving "cell" as a count-fraction, not a literal grouping); (2) retrofit Cell Revolt's compounding-risk clause to reuse Goldenfurt's verified G606 pattern (suspicion cap +1/season, Submit-to-audit escape) rather than inventing parallel escalation math; (3) only then decide KEEP vs REFINE — do not restore MERGE's deletion instructions (retiring §1.3b, the Ledger variant, and the Crisis trigger) without first re-running the executability check against the corrected substrate, since deleting landed text per CLAUDE.md's ED-1094 default is itself a ratifying act if bundled into a routine PR.

### SE-JP2 — Kokudaka cadastral Survey (locked `assessed_base`)

- **Original mechanic:** Periodic (not continuous) assessment locks an `assessed_base` figure; Extract reads the locked figure until the next Survey, making the assessed/actual gap an exploitable surplus (banked hidden growth) or a neglect-detonates-revolt hazard. Landed jointly with HAB-5 at `governance_play_redesign_v1.md` §1.3a (ED-SE-0018), sharing the locked-figure substrate (audit §5.4/S-5).
- **Prior NERS verdict:** **REFINE (confirmed).** Clears N and Omega-d (genuinely new lever, rewards inaction). Fails Q: seeds no §2.3 event-deck card; its declared-Dearth failure mode never wires to `settlement_layer_v30.md` §4.3a (keyed to live Prosperity ≤1, not a stale Assessment); "Assessment" is absent from §1.6's Ledger-family enumeration; no Assessment-vs-Compact precedence; no crisis-break/emergency-resurvey clause for an absolute 8-season cooldown.
- **Re-eval vs built work:** **COLLIDES** — on two independent axes, both worse than the audit's own Q findings. First: `sim/territory/registry.py`'s actual Settlement schema (prosperity/defense/order/legitimacy/popular_support/facility_tier/suspicion/pressure/subnational/npc_ids/ledger/open_needs/deck_state) **has no `assessed_base` field at all**. This is not a missing enumeration bug in §1.6 (as the audit framed it) — it is the absence of the field's entire storage substrate in the built code the port is validated against; SE-JP2 cannot be wired to anything today. Second: the "Assessment-vs-Compact precedence" gap the audit flagged as unresolved is worse than unresolved — it's **Compact-vs-Leverage**. `ledger.py`'s built `TAG_KINDS = {Precedent, Grudge, Debt, Reputation, Leverage}` has no "Compact" entry; the design doc's claim that HAB-5 opens a 5th Ledger family named Compact diverges from the actual 5th built kind, Leverage. SE-JP2 was authored to share substrate with HAB-5's Compact — so it inherits that same unbuilt-tag problem by construction.
- **Changes verdict? Yes** — sharpens REFINE toward a harder prerequisite. The audit treated the missing Dearth wiring and Ledger-enum gap as fixable prose omissions; against the built registry, the fix requires adding a genuinely new schema field (`assessed_base`) to the actual Settlement object and resolving the Compact/Leverage tag-family mismatch, not just editing prose cross-references.
- **Consolidated recommendation:** **reconcile-first.** Do not promote §1.3a's Survey clause to promote-ready until (1) `assessed_base` (or an equivalent stale-valuation field) is added to the built Settlement schema and reconciled with `effective_prosperity` (SUM aggregation) and the province-level floor-average-Order aggregation it would need to interact with at province scale; (2) the Compact-vs-Leverage tag-family collision is resolved — either retitle the design doc's 5th family to match built Leverage, or file a change request against `ledger.py` to add Compact as a genuine 6th kind, loudly, not silently. This is the same reconciliation HAB-5 needs (see cross-reference below); the two should not land as separately-worded fixes.

### SE-JP3 — Za guild patronage lapse

- **Original mechanic:** Extends the settlement Charter tag (§3.3a) with a patron field: guild privileges lapse automatically when the patron's Standing falls, opening a new attack vector (target the patron, not the guild). Landed at `settlement_layer_v30.md` §3.3b (ED-SE-0021).
- **Prior NERS verdict:** **REFINE (overturned: MERGE → REFINE).** The dossier's MERGE (duplicate of ED-FA-0021/Kaochengfa) failed on direct read — different scale, tag family, and resolver; "patron-bound lapse" is a shared shape, not a shared mechanism, and the corpus's own later research treats the shape as meant to recur. Real, narrower defect: Omega-d — patron-collapse lapse silently bypasses §3.3a's Quo Warranto age-scaling (`floor(charter_age_seasons/8)`) for old, prescription-hardened charters, with no age term and no contagion echo.
- **Re-eval vs built work:** **COLLIDES** — the section SE-JP3 explicitly extends, §3.3a's "Charter" tag, is **not among the built `TAG_KINDS`**. Built ledger tag kinds are Precedent/Grudge/Debt/Reputation/Leverage; there is no Charter entry, mirroring the exact same absence the audit already flagged for HRE-5 ("references a non-existent Charter tag"). SE-JP3 was authored on the assumption that §3.3a Charter is real, durable, engine-resolvable tag machinery; against the built ledger it is design-prose with no code-side representation. Separately, the audit's own Omega-d fix (inherit Quo Warranto's `floor(charter_age_seasons/8)` friction) presumes `charter_age_seasons` is a tracked quantity — also absent from the enumerated Settlement schema, which has no age/duration field for any tag.
- **Changes verdict? Yes.** The prior REFINE treated the fix as two single-paragraph prose additions (age-scaling clause, ED-FA-0021 citation). Against the built ledger, the prerequisite is a schema decision: does "Charter" map onto one of the five real `TAG_KINDS` (most plausibly Leverage — a revocable standing privilege — or Precedent), or does it require a genuine new kind? Until that's settled, SE-JP3's whole mechanism (privileges keyed to a tag that doesn't exist in code) is unbuilt.
- **Consolidated recommendation:** **reconcile-first**, then refine. (1) Map §3.3a "Charter" onto the nearest real `TAG_KINDS` entry (Leverage is the strongest fit — a revocable, patron-contingent standing privilege) rather than treating Charter as a de facto 6th family alongside the equally-unbuilt Compact; (2) once mapped, re-derive the age-scaling fix against whatever durability/age tracking the chosen tag kind actually supports in `ledger.py`, since `charter_age_seasons` itself has no confirmed schema home; (3) retain the audit's citation-to-ED-FA-0021 fix as-is — that finding is prose-level and unaffected by the built-substrate gap.

### SE-JP4 — Shi-no-ko-sho status freeze

- **Original mechanic:** A caste-mobility lock trading Order for hard-blocked advancement — a settlement-level toggle suppressing card generation and Order-decay at the cost of freezing caste-crossing Rank Advancement entirely. Queued (`ED-SE-0036`), never authored into canon.
- **Prior NERS verdict:** **CUT (confirmed).** Two independently sufficient failures: (i) Omega-d dominant strategy — unlike every costed sibling (Bind-the-Cells' compounding cost, Survey/Assessment's decay-if-unrefreshed), the freeze's benefit is static, monotonic, and zero-upkeep once active — invoke once, leave on forever; (ii) T-fail — its blanket "any Caste × Rank Advancement crossing" block, with no carve-out, silently nullifies the Warden-of-the-Thread/Riskbreaker/Niflhel escape valves Part 3 *deliberately* authored as caste-critique resistance infrastructure.
- **Re-eval vs built work:** **SUPERSEDED** — built facts do not rescue this item; they compound the existing failure and add a new one. Order is a real built Settlement field (0–5), so the freeze's headline lever is at least wired to something real — but the type-taxonomy drift the built work itself surfaces (`settlement_mgmt_stress_01`'s root finding F1: Village/Fortress-City/Cathedral-City undefined in §1.2, tracked as baseline-audit H3) means the very notion of a settlement-*type*-scoped freeze condition sits on ground the built work has already flagged as undefined. A mechanic gating a permanent, zero-upkeep lock on caste-crossing advancement is exactly the kind of irreversible, hard-to-audit toggle that should not be layered onto a taxonomy the sim's own stress-test found ambiguous enough to cause a runaway-negative regime elsewhere.
- **Changes verdict? No** — still CUT, and more firmly. Nothing in the built substrate (registry schema, Goldenfurt's worked mechanics, or the aggregation formulas) supplies the missing carve-out logic or converts the Omega-d dominance into a real tradeoff; if anything, tying the freeze condition to an undefined type taxonomy (F1/H3) is a new, independent reason not to build it as specified.
- **Consolidated recommendation:** **drop.** Do not author the §1.3a Freeze subsection or the 13th exception column on the §3.2 caste matrix per the prior audit's follow-up; do not revisit pending a taxonomy fix, since fixing F1/H3 would remove the newly-identified aggravating factor but not the underlying Omega-d/T-fail pair that already independently routes this to CUT.

### FA-JP5 — Metsuke/Ometsuke segregated surveillance

- **Original mechanic:** Splits faction surveillance into two target-segregated, non-poolable tracks (own-officers watch vs. vassal-lord watch) — a faction funding one is structurally blind to the other's targets. Queued (`ED-FA-0033`), never authored.
- **Prior NERS verdict:** **DISTILL (confirmed).** The steelman's real point (suzerain-vassal surveillance is a control problem the Suspicion→Recall loop structurally can't reach, since a Jarl is never appointed/recallable) survives, but the proposal fails: no funding cost/decay/clamp specified; the novel half (Vassal-Lord Watch) ties to FA-JP3, itself `needs_jordan`/open; it re-opens ED-SE-0005's deliberate no-new-plumbing narrowing of Investigate, unacknowledged; a live Intel stat + Ministry Competence/Corruption apparatus already occupy adjacent space.
- **Re-eval vs built work:** **RECONCILE** — new, genuine collision the prior audit had no way to see, since it predates the built-substrate read. `sim/territory/registry.py`'s built Settlement schema carries a **single scalar `suspicion` field**, and `goldenfurt_slice`'s G606 finding is a *verified* fix built directly on that single scalar (suspicion capped +1/season, feeding the recall-death-spiral escape). FA-JP5 proposes splitting surveillance into **two independent, non-poolable tracks**. If FA-JP5's "own-officers" track is meant to *be* the built settlement-scale `suspicion` field (plausible, since both gate Recall/Investigate-adjacent behavior), forking it into two tracks would fragment or bypass the exact mechanism G606 already verified as solving the recall spiral — a real risk of quietly breaking working built code. If instead FA-JP5's surveillance is meant as a wholly separate FA-lane Intel resource, it needs to say so explicitly, because as written it reads as a variant of the same suspicion concept the settlement layer already tracks and has already load-bearing-verified a fix for.
- **Changes verdict? Yes** — sharpens DISTILL with a concrete new risk the prior pass didn't have visibility into: this isn't just under-specified (no cost/decay/clamp), it risks colliding with a *working, verified* built mechanism (G606's suspicion-cap escape valve) if the two surveillance tracks are read as forking the same field.
- **Consolidated recommendation:** **distill**, with an added precondition before any promotion — the prior audit's fold-in (own-officer half → §1.4 Suspicion track; aggregate-surveillance half → §7.1 Schattendienst Competence/Corruption funding tradeoff) must explicitly state whether "§1.4 Suspicion track" *is* the built `registry.py` `suspicion` scalar or a distinct FA-lane field. If it is the same scalar, do not fork it into two non-poolable tracks — that would undo the G606-verified single-track fix without a demonstrated replacement; route the vassal-lord-watch half through a genuinely separate resource instead. Any residual target-segregation flavor should land as an Ob-modifier note, not a new resource axis, as the prior audit already specified — and only with an explicit logged supersession of ED-SE-0005 if it touches Investigate.
